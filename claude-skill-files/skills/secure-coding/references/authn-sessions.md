# Authentication, Sessions & Identity

**Core principle:** Buy before you build. Authentication is the most-attacked surface
and the hardest to implement correctly. Production-tested platforms have received far
more adversarial scrutiny than custom implementations.

Load `references/lang/<lang>.md` for language-specific cookie config, JWT, and password
reset patterns. Load `references/cloud-aws.md` or `references/cloud-azure.md` for
managed identity and SSO integration patterns.

---

## Authentication

### Provider First
Prefer a managed identity platform over custom login:
- **AWS:** Cognito, or federate with an IdP via IAM Identity Center
- **Azure:** Azure AD B2C, Entra External ID, or Managed Identity for service-to-service
- **Third-party:** Auth0, Okta
- **Framework built-in:** ASP.NET Core Identity, Django auth, Devise (Rails)
- **Build from scratch:** last resort only; requires AppSec review

### Password Requirements (NIST SP 800-63B)
- Minimum 8 characters; allow at least 64
- Allow all printable ASCII and Unicode; no arbitrary character restrictions
- Do NOT enforce complexity rules (uppercase + number + symbol); they produce weaker,
  more predictable passwords and user workarounds
- Check new passwords against breached-password lists (Have I Been Pwned API or static list)
- Implement account lockout or exponential backoff on failed attempts
- Hash with Argon2id, bcrypt (cost ≥ 12), or scrypt — see `references/secrets-crypto.md`

### Multi-Factor Authentication
- Prefer TOTP (RFC 6238) or FIDO2/WebAuthn over SMS
- SMS is susceptible to SIM swap; use it only if no stronger option is available
- Provide backup recovery codes; hash them like passwords before storage

## Session Management

### Core Rules
- Use the session management built into your framework; never roll your own
- Session IDs: minimum 128 bits, generated by a CSPRNG, unpredictable, no user-meaningful content
- Cookie flags: `HttpOnly`, `Secure`, `SameSite=Strict` (or `Lax` for cross-origin flows)
- Issue a **new session ID** on every privilege change: login, logout, role elevation
- Invalidate sessions **server-side** on logout; never rely solely on cookie deletion
- Set idle timeout and absolute expiry; do not allow sessions to persist indefinitely

### Session Fixation
If the session ID is preserved across login (not regenerated), an attacker who planted
the ID before login can use it afterwards. Always regenerate on login.

## JWTs

JWTs are appropriate for stateless API auth. They are frequently misused.

### Rules
- Algorithm: **RS256** or **ES256** for asymmetric (preferred); **HS256** with ≥ 256-bit key for symmetric
- **Never accept `alg: none`** — explicitly allowlist algorithms before verification
- Validate: `exp` (expiry), `iat` (issued-at), `iss` (issuer), `aud` (audience)
- Access token lifetime: 5–15 minutes
- Refresh tokens: longer-lived, stored `HttpOnly` in a cookie, rotated on use
- Payload is base64-encoded, not encrypted — never put sensitive data in claims

### alg:none Attack
Many JWT libraries historically accepted unsigned tokens when `"alg": "none"` was
specified in the header. Always specify allowed algorithms explicitly in your
validation call. Never default to "accept whatever the token header says."

## OAuth2 / OIDC

- Use **PKCE** for all authorization code flows (server-side apps, SPAs, mobile)
- Validate the `state` parameter on the callback to prevent CSRF
- Validate the `nonce` in ID tokens for OpenID Connect flows
- Never use the implicit flow — it exposes tokens in the URL

## Password Reset

- Token: minimum 256 bits from a CSPRNG
- Store: hash of the token (not the token itself); use constant-time comparison to verify
- Expiry: 15–60 minutes; single-use
- Invalidate all outstanding reset tokens on successful password change
- Never reveal whether an email address exists: use identical response for found/not-found
- Rate-limit reset requests per email address and per IP

## Error Messages

Authentication errors must never reveal which field was wrong:
- Wrong: "Password incorrect for that account"
- Wrong: "No account with that email address"
- Correct: "Invalid username or password"

User enumeration via error messages is a medium-severity finding that enables
targeted attacks and credential stuffing.
